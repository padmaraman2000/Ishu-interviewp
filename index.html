Round 2 – Technical Round

Role-specific and domain questions. Tap to expand answers.

🛢️ SQL / Database
1 Write a query to find duplicate records in a table. ▼
Answer:
SELECT column_name, COUNT(*) as count FROM table_name GROUP BY column_name HAVING COUNT(*) > 1;
In production support, I use this to find duplicate loan applications or duplicate transaction entries that may cause system errors.
2 What is the difference between WHERE and HAVING? ▼
WHERE filters rows before grouping. HAVING filters after grouping (used with aggregate functions like COUNT, SUM).

Example: WHERE status = 'FAILED' filters individual rows. HAVING COUNT(*) > 5 filters groups.
3 Explain INNER JOIN vs LEFT JOIN with an example. ▼
INNER JOIN – Returns only matching rows from both tables.
LEFT JOIN – Returns all rows from the left table + matching rows from right (NULL if no match).

In fintech: To get all loan applications and their disbursement status — use LEFT JOIN so we see applications that were applied but not yet disbursed.
☁️ AWS CloudWatch & Monitoring
4 How do you use AWS CloudWatch in your daily work? ▼
"I use CloudWatch daily for application log monitoring and alert resolution. I check log groups for error patterns — like API timeout errors, failed transaction callbacks, or database connection issues. I also use CloudWatch Alarms to get notified when error rates spike. When an alert fires, I drill into the log streams to find the exact request that failed and correlate it with SQL data to identify root cause."
5 What is the difference between CloudWatch Logs and CloudWatch Metrics? ▼
CloudWatch Logs – Stores raw log data from applications (e.g., error messages, API request logs). You search and filter these to debug issues.

CloudWatch Metrics – Numerical time-series data (e.g., CPU usage, error count per minute). Used for dashboards and alarms. Metrics show "how much" — logs show "what exactly happened."
🔌 API Testing
6 How do you test an API using Postman in production support? ▼
"When a production issue is reported — for example, 'loan disbursement API is failing' — I open Postman, recreate the exact request using the same payload the customer sent. I check:
• Status code (200 OK vs 400/500 errors)
• Response body — error message, field validation failures
• Headers — auth token validity, content type
Then I compare with CloudWatch logs to see if it's a code error or a data error. This helps me give a clear RCA to the dev team."
7 What are HTTP status codes? Name important ones. ▼
200 OK – Success
201 Created – Resource created
400 Bad Request – Client sent invalid data
401 Unauthorized – Auth token missing/invalid
403 Forbidden – No permission
404 Not Found – Resource doesn't exist
500 Internal Server Error – Server-side bug
502/503 – Gateway/service unavailable
🏦 Fintech Domain
8 What is the difference between LOS and LMS? ▼
LOS (Loan Origination System) – Handles the loan application journey: customer application, credit check, approval/rejection, KYC verification, and loan sanction.

LMS (Loan Management System) – Manages the loan after disbursement: EMI schedules, repayment tracking, overdue management, foreclosure, and reporting.

In production support, LOS issues are usually related to application failures or KYC API errors. LMS issues are related to EMI posting, balance mismatches, or repayment failures.
9 What is AML and why is it important in fintech? ▼
"AML stands for Anti-Money Laundering. It's a set of regulations that financial institutions must follow to detect and prevent the use of their systems for illegal money flows. In my role at M2P Fintech's compliance team, I helped maintain records and supported periodic AML reviews. In production support, if there's a transaction flag from the AML system, it gets escalated to compliance immediately and the transaction is held — understanding this helped me triage such tickets correctly."
🔍 Incident Management
10 How do you handle a P1 (critical) production incident? ▼
"My P1 process:
1. Acknowledge immediately – Pick up the ticket, send an acknowledgement to stakeholders within SLA.
2. Triage – Check CloudWatch logs, run SQL queries to understand the scope (how many customers affected, what exactly is broken).
3. Escalate fast – If it's a code issue, I escalate to the dev team with full context: logs, API payload, SQL output.
4. Communicate – Keep stakeholders updated every 15-30 minutes.
5. RCA document – After resolution, document root cause, fix, and preventive action."
11 What is SLA and how do you track it? ▼
"SLA stands for Service Level Agreement — it's the committed response and resolution time for different ticket priorities. For example, P1 might be 1-hour response and 4-hour resolution. I track SLA using Jira dashboards and monitoring the ticket aging in our queue. If a ticket is approaching its SLA breach threshold, I escalate or re-prioritise immediately. I've consistently maintained SLA compliance in my current role."
